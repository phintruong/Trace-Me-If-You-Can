================================================================================
          INVESTIGATOR COPILOT - SYSTEM DESIGN & IMPLEMENTATION PLAN
                          GenAI-Genesis AML Platform
================================================================================

CONTEXT
-------
GenAI-Genesis currently runs a pipeline (CSV -> GNN -> Railtracks agents -> API)
that flags suspicious accounts and displays them on a dashboard. This plan adds
a case management and AI investigation system on top, transforming the platform
into a Palantir-style financial crime investigation copilot.

The existing pipeline already produces everything needed:
  - Graph edges with from/to/amount
  - Per-account risk scores (0-1)
  - Pattern labels (circular, hub, rapid_movement)
  - LLM infrastructure (_call_llm in railtracks_explainer.py)

Storage: JSON files on disk (no database for case data). Each case is a
self-contained JSON file in backend/outputs/cases/.

All work is BACKEND-ONLY (no frontend changes).

PHASE 4: CASE BUILDER SERVICE (core intelligence)
--------------------------------------------------------------------------------

New file: backend/app/services/case_builder.py

This is the most important new file. Contains graph analysis algorithms and
case construction logic.

ALGORITHM: Subgraph Extraction (BFS)
  def extract_subgraph(account_id, graph_edges, account_risk_scores,
                       max_depth=3, risk_floor=0.3, max_accounts=50):
      - Build adjacency dict from graph_edges (both directions)
      - BFS from account_id
      - Only follow edges to accounts with risk_score >= risk_floor
      - Stop at max_depth or max_accounts
      - Returns: set of related account_ids, list of subgraph edges

ALGORITHM: Role Assignment
  def assign_role(account_id, subgraph_edges) -> str:
      - Compute inflow, outflow, in_degree, out_degree from subgraph edges
      - total_degree >= 10         -> "hub"
      - outflow > 0, inflow == 0   -> "source"
      - inflow > 0, outflow == 0   -> "sink"
      - similar in/out amounts     -> "mule" (pass-through)
      - else                       -> "unknown"

ALGORITHM: Money Path Discovery
  def find_money_paths(source_id, edges, max_depth=6) -> list[list[str]]:
      - DFS from source following outgoing edges
      - Track visited per-path to detect cycles
      - Return paths of length >= 3
      - Used to identify layering chains

ALGORITHM: Priority Computation
  def compute_priority(risk_score, pattern_count, related_count) -> str:
      - risk >= 0.9 or patterns >= 3  -> CRITICAL
      - risk >= 0.7 or patterns >= 2  -> HIGH
      - risk >= 0.5                   -> MEDIUM
      - else                          -> LOW

MAIN ORCHESTRATOR:
  def build_case(account_id, account_risk_scores, graph_edges, account_patterns) -> str:
      1. Extract subgraph via BFS
      2. Assign roles to each account in subgraph
      3. Collect patterns from existing account_patterns + detect new (layering paths)
      4. Compute priority
      5. Generate case_id: "CASE-{YYYYMMDD}-{uuid4[:8]}"
      6. Build full case dict
      7. Generate AI summary via copilot_agent.generate_case_summary()
      8. Save via case_store.save_case()
      9. Return case_id

  def generate_cases_from_pipeline(flagged_accounts, account_risk_scores,
                                    graph_edges, account_patterns,
                                    threshold=0.7) -> list[str]:
      - For each flagged account above threshold:
          - Skip if account already has an OPEN case (dedup)
          - Call build_case()
      - Return list of case_ids

PHASE 7: COPILOT AGENT
--------------------------------------------------------------------------------

New file: backend/app/services/copilot_agent.py

  def ask_copilot(case_id: str, question: str) -> dict:
      1. Load full case from case_store.get_case()
      2. Build context string from all case data
      3. Detect question type for prompt routing:
         - "summarize"/"summary"           -> summary generation prompt
         - "explain"/"why"/"chain"         -> transaction chain explanation
         - "next"/"recommend"/"steps"      -> investigation next-steps
         - "sar"/"report"/"filing"         -> SAR-ready narrative (FinCEN format)
         - default                         -> general case Q&A
      4. Call call_llm() with case-aware system prompt
      5. Return {"answer": str, "context_used": ["case_data", "patterns", ...]}

  def generate_case_summary(case: dict) -> str:
      - Build context from case data
      - Call call_llm() with summary-focused prompt
      - Return summary text (saved into case["ai_summary"])

System prompt template:
  You are an AML investigation copilot. Case data:
  - Case: {case_id}, Status: {status}, Risk: {risk_score}
  - Primary Account: {primary_account_id}
  - Related Accounts ({n}): {accounts with roles}
  - Patterns: {pattern descriptions}
  - Timeline: {n_events} events, {timeline_summary}
  - Notes: {investigator_notes}

  Answer the investigator's question. Be specific - cite account IDs and amounts.
  If asked for a SAR narrative, use FinCEN SAR format.

PHASE 8: TIMELINE BUILDER
--------------------------------------------------------------------------------

New file: backend/app/services/timeline_builder.py

  def build_timeline(case: dict, graph_edges: list[dict]) -> list[dict]:
      1. Collect all edges involving case account IDs from graph_edges
      2. Sort by available timestamp or amount as proxy
      3. Classify events:
         - Account with only outgoing edges -> "source_deposit"
         - One sender -> multiple receivers  -> "split" (fanout)
         - Multiple senders -> one receiver  -> "recombination" (fanin)
         - Transfers between case accounts   -> "layering_transfer"
         - Account with only incoming edges  -> "terminal_withdrawal"
         - Default                           -> "transfer"
      4. Assign significance:
         - Amount > 2x median -> "high"
         - Risk > 0.8         -> "high"
         - Otherwise          -> "medium" or "low"
      5. Return ordered list of TimelineEvent dicts

Note: graph_edges from build_graph_from_raw() carry from/to/amount but no
timestamps. Timeline classifies primarily by flow structure (fanout/fanin/cycles).
